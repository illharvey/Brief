# Phase 5.1: Settings Page - Context

**Gathered:** 2026-03-04
**Status:** Ready for planning

<domain>
## Phase Boundary

A `/dashboard/settings` page where logged-in users can view and update their topics and delivery time. Adds a full sidebar layout to the dashboard shell. No new capabilities beyond PREF-03: update topics and delivery time from the web app.

</domain>

<decisions>
## Implementation Decisions

### Topics editing UX
- Tag pills (removable chips) — use the existing `pill active/inactive` pattern from the landing page design
- Add topics via freeform text input: type + Enter or comma to create a new pill
- Duplicate prevention only — no minimum/maximum topic count enforced
- Suggestions are **in scope**: a curated, hardcoded list of ~20-30 popular topics
- Suggestions displayed as **static clickable chips** below the input field (always visible, not a dropdown)
- Topics already added as pills are **hidden** from the suggestion chips
- Show all ~20-30 suggestions at once (no "show more" collapse)
- Suggestions are distinct from PREF-04 (AI-adjacent suggestions) — these are a fixed curated list

### Delivery time input
- Dropdown with preset slots: **6 AM – 10 PM, 1-hour increments** (16 options)
- Display user's detected timezone next to the selector (e.g. "Delivery time (GMT+0)" or "7:00 AM (BST)")
- Detect timezone client-side; store the time preference alongside the timezone string
- Dispatch uses the stored timezone — times are always local to the user

### Save behavior
- **Single "Save preferences" button** for the whole page (not per-section)
- On success: **toast notification** — "Preferences saved"
- On failure: **error toast with retry** — "Failed to save. Try again."
- No unsaved-changes indicator or navigation warning — keep it simple

### Page layout & navigation
- **Full sidebar layout** added to the dashboard shell (updates `dashboard/layout.tsx`)
- Sidebar uses **dark espresso (`#1C0F05`) background** with beeswax-colored "Brief." logo in Playfair Display
- Sidebar structure (extensible for future phases):
  - Top: "Brief." logo
  - Nav: "Briefings" (muted/placeholder — active in Phase 6) + "Settings" (active, beeswax accent)
  - Bottom: user email + "Sign out" button
- Main content area: **`--milk` (#FDF9F0) background** with grain texture overlay (matches landing page)
- Settings page heading: Playfair Display 900 (editorial serif) + DM Sans subtitle below it

### Design system (locked from HTML reference)
- Colors: `--milk: #FDF9F0`, `--espresso: #1C0F05`, `--beeswax: #C1522A`, `--beeswax-deep: #A33E1A`, `--warm-brown: #5C3D1E`, `--muted: #7A5C3A`, `--steam: #F0E8D5`, `--cream: #FAF4E8`
- Fonts: Playfair Display (headings), DM Sans (UI/labels/body), Libre Baskerville (long-form body only)
- Border-radius: 2px on buttons and inputs (sharp, editorial)
- Grain overlay: fixed pseudo-element, `opacity: 0.4`, SVG fractalNoise — persists across all dashboard pages

### Claude's Discretion
- Exact spacing, padding, and typography scale inside the settings form
- How to handle the Tailwind implementation of the CSS variables (may need to extend tailwind config)
- Whether to use shadcn/ui's `Select` component for the time dropdown or a custom implementation
- Toast library choice (sonner, react-hot-toast, or shadcn toast)
- Tag pill input component approach (custom or cmdk/react-tag-input based)

</decisions>

<specifics>
## Specific Ideas

- The landing page HTML is the canonical design reference. The dashboard should feel like a natural extension of that visual language — same grain, same palette, same type scale.
- The `pill active` style from the landing page (beeswax background, espresso text, 20px border-radius) is the exact style to reuse for selected topic pills in the settings input.
- The `topic-card` hover style (beeswax border-left accent, subtle lift) can inform the suggestion chip hover state.
- The sidebar "Brief." logo should match the `nav-logo` style: Playfair Display, font-weight 900, with the period in beeswax-deep (`#A33E1A`).

</specifics>
