## Summary

Phase 5.1 is a dashboard feature phase — not a new infrastructure phase. Everything needed already exists in the codebase: the DB tables (`userTopics`, `deliveryPreferences`), the server actions (`saveTopicsAction`, `saveDeliveryPreferenceAction`), the onboarding components (`TopicInput`, `TimePicker`), and the DB query patterns (Drizzle ORM). The task is to: (1) add the design system tokens to globals.css + load the new fonts, (2) replace the dashboard shell with a full sidebar layout, and (3) build the `/dashboard/settings` page using or adapting the existing components.

The `preferencesUrl` in `email.ts` is already hardcoded to `${APP_URL}/dashboard/settings`. This means the "Preferences" link in briefing emails is already correct — it will stop returning 404 the moment the route exists. No email template changes are needed.

The time picker must be replaced: the current `TimePicker` uses a freeform `<input type="time">` but the decision locks in a dropdown with 16 preset slots (6 AM–10 PM, 1-hour increments). The existing `saveDeliveryPreferenceAction` already handles timezone storage correctly (IANA string) and works for both onboarding and post-onboarding contexts. The `saveTopicsAction` also already handles the update path (delete-then-insert transaction). Both actions are session-aware via `resolveUserId()`, which checks session first.

**Primary recommendation:** Build the settings page as a Next.js Server Component that reads current preferences, pass them to a Client Component form that reuses the existing server actions, add Sonner for toasts (available via `shadcn add sonner`), and add the design system tokens to `globals.css` via Tailwind v4 `@theme` extension.

### Pattern 1: Server Component reads DB, passes to Client Component form

**What:** The settings page (`settings/page.tsx`) is a Server Component that calls `auth()` and queries the DB directly. It passes the current preferences as props to the Client Component form.

**When to use:** Always — avoids an extra API round-trip on page load, uses the session already established by `dashboard/layout.tsx`.

```typescript
// src/app/dashboard/settings/page.tsx — Server Component
import { auth } from "@/lib/auth"
import { redirect } from "next/navigation"
import { db } from "@/lib/db/client"
import { userTopics, deliveryPreferences } from "@/lib/db/schema"
import { eq } from "drizzle-orm"
import { SettingsForm } from "./settings-form"

export default async function SettingsPage() {
  const session = await auth()
  if (!session?.user?.id) redirect("/login")

  const userId = session.user.id

  // Parallel queries — both are needed before render
  const [topicRows, deliveryPref] = await Promise.all([
    db.query.userTopics.findMany({
      where: eq(userTopics.userId, userId),
      columns: { topic: true },
    }),
    db.query.deliveryPreferences.findFirst({
      where: eq(deliveryPreferences.userId, userId),
      columns: { deliveryTime: true, timezone: true },
    }),
  ])

  const currentTopics = topicRows.map((r) => r.topic)
  const currentDeliveryTime = deliveryPref?.deliveryTime ?? "08:00"
  const currentTimezone = deliveryPref?.timezone ?? ""

  return (
    <SettingsForm
      initialTopics={currentTopics}
      initialDeliveryTime={currentDeliveryTime}
      initialTimezone={currentTimezone}
    />
  )
}
```

### Pattern 2: Client Component form with useActionState + Sonner

**What:** The `SettingsForm` client component holds topics/time state, submits to existing server actions, and shows toasts via Sonner on success/failure.

**When to use:** Whenever a form needs optimistic UI or toast feedback after a server action.

```typescript
// src/app/dashboard/settings/settings-form.tsx — Client Component
"use client"
import { useState, useActionState, useEffect } from "react"
import { toast } from "sonner"
import { saveTopicsAction, saveDeliveryPreferenceAction } from "@/actions/preferences"
import { TopicInput } from "@/components/ui/topic-input"
import { Button } from "@/components/ui/button"
// ... select imports for time dropdown

// Wrapper combines both actions — settings page saves all at once
async function saveSettingsAction(prevState: unknown, formData: FormData) {
  const topicsResult = await saveTopicsAction(formData)
  if (topicsResult.error) return topicsResult
  const deliveryResult = await saveDeliveryPreferenceAction(formData)
  return deliveryResult
}

interface SettingsFormProps {
  initialTopics: string[]
  initialDeliveryTime: string
  initialTimezone: string
}

export function SettingsForm({ initialTopics, initialDeliveryTime, initialTimezone }: SettingsFormProps) {
  const [topics, setTopics] = useState<string[]>(initialTopics)
  const [deliveryTime, setDeliveryTime] = useState(initialDeliveryTime)
  const [timezone] = useState(initialTimezone) // read from DB, refreshed client-side
  const [state, formAction, isPending] = useActionState(saveSettingsAction, null)

  useEffect(() => {
    if (!state) return
    if ("success" in state && state.success) {
      toast.success("Preferences saved")
    } else if ("error" in state) {
      toast.error("Failed to save. Try again.")
    }
  }, [state])

  return (
    <form action={formAction} className="space-y-8">
      <input type="hidden" name="topics" value={JSON.stringify(topics)} />
      <input type="hidden" name="timezone" value={timezone} />
      <input type="hidden" name="deliveryTime" value={deliveryTime} />
      {/* TopicInput + time Select go here */}
      <Button type="submit" disabled={isPending}>
        {isPending ? "Saving..." : "Save preferences"}
      </Button>
    </form>
  )
}
```

### Anti-Patterns to Avoid

- **Reading preferences in the Client Component via useEffect + fetch:** Use the Server Component pattern (Pattern 1) to read from DB on initial render. Client-side fetch adds a loading flash and round-trip.
- **Two separate Save buttons (one per section):** Decision is locked — single "Save preferences" button for the whole page.
- **Calling `resolveUserId()` with cookie fallback in settings:** The settings page is post-onboarding; the user is always authenticated via session. `saveTopicsAction` and `saveDeliveryPreferenceAction` already use `resolveUserId()` which checks session first — this is correct.
- **Adding `--muted` to Tailwind theme:** This name conflicts with shadcn's `--muted` token (the gray muted color). Use `--color-brief-muted` or `--color-warm-muted` for the `#7A5C3A` value.
- **Storing timezone from initial DB value without re-detecting client-side:** The stored timezone may be stale if the user has moved. Re-detect with `Intl.DateTimeFormat().resolvedOptions().timeZone` on mount (as the existing `TimePicker` does) and use the fresh value for the save.
- **Breaking the existing `dashboard/page.tsx`:** The layout change must preserve `dashboard/page.tsx` working — it currently renders standalone; after adding the sidebar layout it will render inside the sidebar shell. This is correct behavior.

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Toast notifications | Custom toast state + portal | Sonner via `shadcn add sonner` | Race conditions with multiple toasts, accessibility, positioning, dismiss logic are all handled |

| Font optimization | Self-hosted font files | `next/font/google` | Automatic font subsetting, zero layout shift, served from same origin |

**Key insight:** The most important work in this phase is UI assembly, not new infrastructure. The server actions, DB schema, and component primitives are all in place.

## Common Pitfalls

### Pitfall 1: `--muted` CSS variable name collision
**What goes wrong:** Adding `--color-muted: #7A5C3A` (the Brief warm-muted color) to `@theme` shadows shadcn's own `--muted` variable (`oklch(0.967 0.001 286.375)` — the light gray). All shadcn components using `bg-muted` or `text-muted-foreground` will render with the wrong color.
**Why it happens:** Both shadcn and the Brief design system want to use "muted" but mean different things.
**How to avoid:** Use a namespaced name for Brief's color: `--color-brief-muted` or `--color-warm-muted`. Do NOT use `--color-muted`.
**Warning signs:** shadcn card backgrounds or secondary buttons turning warm-brown during testing.

### Pitfall 2: `dashboard/page.tsx` sign-out form breaking after sidebar layout
**What goes wrong:** `dashboard/page.tsx` currently embeds `<form action={signOutAction}><Button>Sign out</Button></form>` — this becomes redundant once the sidebar adds its own sign-out. If left in, there will be two sign-out buttons.
**Why it happens:** The page was a placeholder with temporary nav; the sidebar supersedes it.
**How to avoid:** Remove the sign-out form from `dashboard/page.tsx` when adding the sidebar (the sidebar will carry sign-out at the bottom). The page content should just show the Phase 6 placeholder text.

### Pitfall 3: TopicInput pre-population — `value` prop initializes but state is local
**What goes wrong:** `TopicInput` is a controlled component accepting `value` and `onChange`. If `useState([])` is used in `SettingsForm` without seeding from `initialTopics`, the existing topics won't show on load.
**Why it happens:** Uncontrolled initialization — `useState` only reads its argument once.
**How to avoid:** Initialize `useState<string[]>(initialTopics)` in `SettingsForm`, not `useState([])`.

### Pitfall 4: Timezone re-detection overwriting stored preference unexpectedly
**What goes wrong:** If the client-side `useEffect` immediately overwrites the timezone from DB with the browser's detected timezone, a user who saved in a different timezone (e.g., during travel) will silently have their timezone reset on page load before they press Save.
**Why it happens:** The existing `TimePicker` always runs `Intl.DateTimeFormat()` on mount and calls `onTimezoneChange`.
**How to avoid:** In the settings form, use the stored timezone as the initial value. Re-detect on mount only if the stored timezone is empty. Or: always re-detect (the most accurate), but show the current detected timezone clearly so users are aware. Since delivery accuracy is important, re-detecting on mount is correct — but be aware it's a silent change until Save is pressed.

### Pitfall 5: toast.success not showing if state reference doesn't change
**What goes wrong:** `useEffect(() => { ... }, [state])` — if the server action returns the same object reference on consecutive saves, React's effect won't re-run.
**Why it happens:** Server actions return new objects on each call, so this is not a real risk in practice. But if `prevState` is accidentally returned unchanged, the effect won't fire.
**How to avoid:** Always return a fresh object from the server action wrapper. The pattern `return deliveryResult` (which is a fresh object from the action) is correct.

### Pitfall 6: The `saveSettingsAction` wrapper must call both sub-actions correctly
**What goes wrong:** Topics and delivery preference are separate actions in `preferences.ts`. A combined wrapper must pass the same `formData` to both. If delivery action is called without `deliveryTime` hidden input, it will fail validation.
**Why it happens:** The settings form has one `<form>` but needs to call two separate server actions.
**How to avoid:** Include hidden inputs for `topics`, `deliveryTime`, and `timezone` in the form. The combined wrapper calls `saveTopicsAction(formData)` first (fail-fast), then `saveDeliveryPreferenceAction(formData)`.

### Pitfall 7: The `preferencesUrl` is already set in email.ts
**What goes wrong:** Nothing — but it's easy to forget that the email link fix is automatic. The dispatch code already hardcodes `${APP_URL}/dashboard/settings`. Once the route exists, the link works. No email code changes needed.
**Why it happens:** Phase 5 was written with this phase in mind.
**How to avoid:** Do not touch `email.ts` or the briefing email template in this phase.

## Open Questions

1. **Onboarding `TimePicker` — update or leave?**
   - What we know: The locked decision specifies a dropdown for the settings page. The onboarding delivery page (`/signup/delivery`) still uses `<input type="time">`.
   - What's unclear: Whether the onboarding should also move to the dropdown now, or remain as-is.
   - Recommendation: Out of scope for Phase 5.1 — the decision only specifies the settings page. Leave onboarding unchanged.

2. **Border-radius: 2px — applying to shadcn components**
   - What we know: The design system specifies 2px border-radius on buttons and inputs. Tailwind's `rounded` generates `border-radius: 0.25rem` (4px). The `--radius` variable in globals.css is `0.625rem`.
   - What's unclear: Whether to override the global `--radius` shadcn variable (which affects all components) or apply `rounded-sm` (4px, close but not 2px) or use `[border-radius:2px]` arbitrary values.
   - Recommendation: Apply `style={{ borderRadius: "2px" }}` or `rounded-[2px]` (Tailwind v4 arbitrary) on the specific settings form elements. Do NOT change the global `--radius` to avoid breaking other shadcn components in the auth flow.

3. **`dashboard/page.tsx` sign-out button removal**
   - What we know: Currently embeds a sign-out form as placeholder navigation. The sidebar will add sign-out at bottom.
   - What's unclear: Whether the planner should update `dashboard/page.tsx` in this phase or leave the duplicate.
   - Recommendation: Remove the sign-out form from `dashboard/page.tsx` when adding the sidebar layout — it is superseded. Replace with just the Phase 6 placeholder text.
